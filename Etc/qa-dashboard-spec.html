01 · Product Identity & Design Philosophy

개요

운영자가 매일 첫 커피와 함께 여는 대시보드. 분기 임원 리뷰에서 열리는 리포트. 매달 장애 복기에서 펼쳐지는 타임라인. 이 모든 일상이 하나의 제품 안에 담긴다.

06 · Write-Back Flow · State Transition

데이터 파이프라인

외부 원천 데이터는 5단계를 거쳐 UI에 도달. 사용자 액션은 같은 경로를 역방향으로 타고 들어가 3중 게이트를 통과해 상태를 변경한다.

상태 변경 예시 · 퀘스트 승인
STEP 01사용자 "승인" 클릭

UI에서 운영자가 승인 액션을 트리거.

STEP 02React Hook → POST

POST /api/quest/{id}/approve 요청 발신.

STEP 03Server handle_admin_approve()

서버 핸들러가 라우팅·권한 검증 후 DB 트랜잭션 진입.

STEP 04quest_db SQLite 트랜잭션

version check · FSM 검증 → UPDATE + _add_history → COMMIT. 3중 게이트가 여기서 작동.

STEP 05JSON 응답 + UI refresh

{ok:true, version:N+1} 반환 → UI 갱신 + 토스트 알림.

3중 게이트 (Triple Gate)
Gate 1 · Optimistic Lock
낙관적 락
version 필드로 동시 수정 conflict 감지. 사용자 A·B가 같은 레코드를 수정하면 후착 트랜잭션은 자동 거부된다.
Gate 2 · FSM
상태 전이 검증
_check_transition이 퀘스트/에피소드별 허용 전이를 검증. 허용되지 않은 상태 이동은 400 반환 + UI 토스트 경고.
Gate 3 · Audit Log
히스토리 기록
모든 상태 변경은 _add_history로 한글 detail 기록(예: "마감일: A → B"). 감사 추적 및 운영자 리뷰용.
Appendix · Glossary

용어 사전

퀘스트 3계층 협업 모델과 시스템 핵심 용어.

용어설명
Episode실(室) 단위 조직 목표이자 최상위 목표. 여러 Quest를 하나의 성과 흐름으로 묶는 상위 개념(분기/반기 단위). 에피소드가 닫히면 모든 Quest 진행률이 기여도로 환산된다.
Quest팀/개인 단위 실행 목표. Episode 달성을 위한 구체 작업 단위로 담당자·기한·상태·진행률 보유. 일반적으로 2~4주 사이클.
SubQuestQuest를 구성하는 실행 가능한 작은 작업 단위. 체크리스트 형태로 관리(1~3일 단위).
Guild특정 목표로 조직된 그룹. 국가·지역·프로젝트 단위 편성. 멤버·포인트·랭킹으로 협업·기여도 집계.
FSMFinite State Machine — 퀘스트/대여 상태 전이 검증.
PRHProduct Release Hub (Market Review) — 마켓 검수.
PBIQA 품질 지표 엑셀 (SharePoint 저장).
MTTRMean Time To Repair — 장애 평균 복구 시간.
Hotfix라이브 긴급 패치.
Optimistic Lockversion 필드로 동시 수정 conflict 감지.
WALWrite-Ahead Logging — SQLite 동시성 향상 모드.
TS2304TypeScript "Cannot find name" 에러 (import 누락).
