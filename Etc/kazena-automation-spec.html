Game QA Automation System — 설계 기획서

스토브 런처에서 카제나 게임을 자동 실행하고, 화면 인식 기반으로 BVT(빌드 검증 테스트)를 수행하여 결과 리포트를 자동 산출하는 시스템. 다수의 QA 담당자가 각자의 PC에서 동일한 자동화 환경을 실행할 수 있는 범용 구조를 목표로 한다.

02 · Implementation Roadmap

구현 순서

의존성 root부터 순차 구현. 사전 지식 베이스 스키마가 모든 후속 컴포넌트의 전제조건이다.

원칙: "보이는 것부터"가 아닌 "의존성 root부터". 사전 지식 베이스 스키마 없이 TC 엔진을 먼저 만들면 재작업 확정. 에픽세븐 베이스를 카제나로 이식하는 작업이 critical path 최상단.
STEP 01 사전 지식 베이스 스키마 정의 P0 · 최우선

에픽세븐 사전 지식 베이스 구조를 카제나로 이식. TC 실행 엔진·인식 파이프라인·리포트 모두 이 스키마에 의존하므로 최우선 확정 필요. 사전 지식 베이스(KB, Knowledge Base)란 AI가 게임 화면을 인식할 때 참조하는 인게임 요소 데이터셋이다.

UI 요소 테이블 — 아이콘 명칭, 버튼 텍스트, 일러스트 ID 정의
화면 테이블 — 로비, 전투, 상점, 설정 등 화면 ID
액션 테이블 — 클릭, 입력, 스와이프, 대기 타입 정의
검증 테이블 — 화면 진입 확인, 텍스트 존재, 재화 증감 조건
캐릭터/재화 분류 — 전투원·파트너 분리, 재화 소모처 정의
STEP 02 브랜치(채널) 선택 구현 P0

브랜치 = 스토브 런처에서 접속할 서버 환경(Dev / QA / Stage 등)을 선택하는 채널. 게임 접속 전 런처에서 어느 서버로 접속할지 선택하는 단계로, 테스트 수행 전 가장 먼저 실행되어야 한다.

런처 실행 → 브랜치 선택 UI 화면 탐지
pyautogui로 해당 브랜치 선택 및 게임 실행 자동화
브랜치 식별 방식 결정 (텍스트 탐지 / UI / 런처 설정파일)
STEP 03 TC 실행 엔진 구현 P0

엑셀 TC를 파싱하여 사전 지식 베이스를 룩업하며 시나리오를 순차 실행하는 핵심 엔진. Action/Verify 추상 레이어로 pyautogui 직접 의존성 캡슐화.

TC DSL 정의 — 스텝 문법 (Action / Verify / Wait / Capture)
엑셀 파서 — 사용자 업로드 TC → 내부 스텝 IR 변환
인식 파이프라인 — 텍스트→아이콘→이미지 순서로 탐지
실행 상태머신 — Idle / Running / Paused / Failed / Aborted
STEP 04 BVT 시나리오 1개 end-to-end P1

로그인 → 로비 진입을 최소 시나리오로 전체 파이프라인 연결 검증. 성공 후 결제·구출 등 시나리오 확장.

로그인 시나리오 — ID/PW 입력 → 서버 선택 → 로비 진입 확인
실패 캡처 — 스텝 실패 시 스크린샷 + 영상 클립 자동 저장
STEP 05 리포트 산출 + 사용자 UI P1

TC ID 매핑 결과 리포트 HTML/PDF 생성. 사용자가 테스트를 설정하고 실행하며 결과를 확인하는 웹 인터페이스 구축.

리포트 — PASS/FAIL/SKIP 요약 + 실패 스크린샷 + 영상 클립 링크
웹 UI — 설정 / 실행 / 결과 3개 주요 화면
환경 진단 — 해상도/DPI/관리자권한/카제나 설치경로 자동 체크
STEP 06 배포 + 히스토리 관리 P2

다수 PC 배포 파이프라인 (PyInstaller + auto-update). 테스트 히스토리 DB 및 전 빌드 대비 회귀 비교.

PyInstaller .exe 빌드 + 자동 업데이트 메커니즘
히스토리 DB + 빌드간 회귀 비교 뷰
사전 지식 베이스 관리 — QA 직접 편집/갱신 인터페이스

05 · Deployment Strategy

배포 방식 비교

약 3명 개발자 유지보수 + 다수 QA PC 독립 실행 조건 기준. exe 단독 vs 하이브리드 장단점.

기준 .exe 단독
PyInstaller		 하이브리드 (권장)
로컬 에이전트 + 웹 대시보드

07 · Risk Register

리스크 등록부

● HIGH — 기술 한계
pyautogui DirectX 풀스크린 한계
카제나가 DirectX 풀스크린 모드로 실행될 경우 화면 캡처 및 좌표 제어 오작동 가능. 안티치트 소프트웨어와 충돌 가능성도 존재.
대응: PoC 단계에서 풀스크린 vs 창 모드 동작 우선 검증. 창 모드 강제 실행 옵션 확인. pywin32(Win32 API) 병행 검토.
● HIGH — 보안
테스트 계정 ID/PW 보안
평문 저장 또는 로그/스크린샷 노출 시 계정 탈취 위험. 결제 TC에서 실결제 방지 실패 시 운영 사고 발생.
대응: Windows Credential Manager 암호화 저장 필수. 로그·스크린샷 ID/PW 자동 마스킹. 결제 TC = 샌드박스 환경만. 실결제 계정 접근 코드 경로 완전 차단.
● MEDIUM — 유지보수
게임 업데이트 시 지식 베이스 재학습 공수
라이브 빌드 업데이트로 아이콘·레이아웃 변경 시 이미지 fallback 소재 재캡처 필요. 업데이트마다 지식 베이스 갱신 공수 발생.
대응: 지식 베이스에 "이미지 버전 + 마지막 확인 빌드" 필드 도입. QA가 직접 편집할 수 있는 관리 UI 필수 구현.
● MEDIUM — 호환성
해상도 / DPI 스케일링 의존성
pyautogui는 화면 해상도와 Windows DPI 설정에 민감. QA 담당자마다 PC 환경이 다르면 좌표 오차 및 탐지 실패 발생.
대응: 실행 전 환경 진단으로 해상도/DPI 자동 체크. 표준 테스트 PC 프로파일 정의 (1920×1080, 100% DPI). 비표준 환경 경고 출력.
● MEDIUM — 범위
PoC 일반화 비용 과소평가
현재 "하드코딩 성공" 상태에서 TC 기반 추상화로 전환 시 실제 공수가 예상보다 클 수 있음.
대응: TC 1건(로그인 시나리오) 완전 추상화 스파이크를 먼저 수행. 실측 공수 기반으로 전체 일정 재산정 후 확장.
● LOW — 운영
지식 베이스 유지보수 주체 부재
개발자만 지식 베이스를 수정할 수 있는 구조가 되면 게임 업데이트마다 개발자 의존 → 도구 사장 위험.
대응: QA 담당자가 직접 지식 베이스를 편집할 수 있는 웹 UI 구현. 개발자 없이 갱신 가능한 구조 필수.
